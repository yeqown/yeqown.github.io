    手搓一个列式数据库

    March 31, 2026
    Database
    Database, OLAP, Columnar Storage

    列式数据库与行式数据库的最大的区别就是数据的存储方式不同，存储方式其实就是说，数据在磁盘上的存储方式不同。传统的行式数据库常用于 OLTP (Online Transaction Processing) 场景，在这个场景下需要频繁的进行数据的插入、更新、删除操作，操作的对象往往是单行数据。而列式数据库常用于 OLAP (Online Analytical Processing) 场景，对于数据的聚合查询更为常见，往往需要扫描某一列的大量数据进行计算。

    这里列举的优化原则只是冰山一角，仅用于说明两种数据库最显眼的差异。

    使用行式数据库过程中，最简单常见的优化原则就是 尽可能命中索引、降低 B+ 树的高度、减少扫描的行数，如：

    • 优先对区分度高的列建立索引
    • 覆盖索引（索引中包含查询所需的所有列，避免回表）
    • 索引下推
    • 最左前缀匹配原则
    • 避免使用函数或者隐式类型转换（如：where date(create_time) = '2022-01-01'），会导致索引失效
    • 避免在索引列上使用 !=、<> 等操作符，会导致索引失效
    • 避免深度分页
    • 分库分表（提出这一优化方向，也是基于单表数据量过大，索引维护的开销会增加，性能也会退化）
    • 等等…

    减少扫描行数这一思路对于列式数据库同样适用（如分区裁剪），但是列式数据库还有其他优化原则，那就是 减少列，如：

    • 行存特性（如果是点查询，列数据库 IO 反而会增加，这一点和行式数据库正好相悖）
    • 只读取查询涉及的列（行存也提倡避免 SELECT *，但由于行存以行为单位读取磁盘，主要减少的是网络传输量而非磁盘 I/O；而列存中每列独立存储，少读一列就直接少一份磁盘 I/O）
    • 等等…

    当然任何数据库的优化，都逃不开减少 I/O这一核心目的。说得更白话一点，如果有一种完美的存储介质，它没有 I/O 延迟，也不会丢失数据，那么这些优化也就不再需要了。

    1. 列式数据库有哪些组件#

    从 ClickHouse 的架构图来看，列式数据库包含以下核心组件：

    • 查询处理层：解析入站查询、构建并优化逻辑与物理查询计划，然后执行
      • SQL Parser
      • SQL Planner
      • Physical Plan Builder
      • Plan Executor
    • 存储层：由不同的表引擎组成，这些表引擎封装了表数据的格式和位置
      • MergeTree* Family Tables Engines
      • Special-Purpose Tables Engines
        • Dictionary
        • Memory
        • Distributed (Data Sharding)
    • 集成层：用于与外部系统进行双向数据交换的虚拟表引擎，例如 PostgreSQL、MySQL、Kafka、RabbitMQ、Redis、Iceberg、S3 等
      • Virtual Tables Engines
    • 正交组件：提供辅助功能
      • Thread pools
      • Caches
      • RBAC
      • Backups
      • Monitoring
    • 访问层：通过不同协议管理用户会话并与应用程序通信
      • User Session
      • Wire protocols

    我们进一步丢弃掉分布式特性、集成和监控，只关注核心组件：

    层 组件 职责
    查询处理层 SQL Parser 解析 SQL 语句，生成抽象语法树 (AST)
    查询处理层 Query Engine 查询计划生成、优化和执行
    存储层 MergeTree Engine 列式存储格式、数据分块、压缩编码、稀疏索引、Data Skipping Index、数据读写

    1.1 列式数据库的存储引擎#

    列式数据库有如下特点：

    • 压缩率高（一般来说每一列的数据都会有重复，尤其是区分度低的列）
    • 大数据集对列的聚合计算更快（只用读取特定列，减少 IO）
    • 按列过滤高效（建立稀疏索引，来跳过不需要的块）
    • 并行度高（得益于列式存储，每一列的数据都是独立的，可以并行处理；再结合表分片和数据分块）
    • insert / update 性能不佳
    • 点查询 / 读取行数据 性能不佳

    1.1.1 磁盘格式#

    每个 MergeTree* 表引擎的表都被组织为一组不可变的表分片，每当一批行数据插入到表中时，会创建一个分片。为了保持每个表的分片数量较少，后台合并任务会定期将多个小分片合并成一个大分片，直到达到可配置的分片大小。

    行数据插入有两种模式：

    • 同步模式：每条 INSERT 创建一个新的分片（因此建议客户端使用批量 INSERT，降低合并开销）
    • 异步模式：将同一个表的 INSERT 缓存在一起，达到一定数量后，才创建新的分片

    在磁盘上，一个分片对应一个目录，其中包含每一列的一个文件。一个分片的行会在逻辑上进一步划分为若干组，每组包含 8192 条记录，称为 granule。granule 表示 ClickHouse 中扫描和索引查询的最小不可再分数据单元。

    但是 granule 只是逻辑概念，在物理上，磁盘数据是在 chunk（块）级别进行的，chunk 将某一列中多个相邻的 granule 组合在一起。一个块上的 granule 数量不是固定的，因为一个块有大小限制，默认是 1MB，受到该列的数据类型和分布影响。块还会被压缩用来减小大小和 IO 成本。

    为了在存在压缩的情况下仍然能够快速随机访问单个 granule，ClickHouse 会为每一列存储一个映射，用于将每个 granule 的 ID 关联到其所在压缩块在该列文件中的偏移，以及该 granule 在未压缩块中的偏移。

    这其实就是一个索引，用于快速查找每个 granule 在存储中的位置。

    列还可以进一步采用字典编码，比如 LowCardinality(T) 使用整数 ID 替换原始列值，Nullable(T) 则为列 T 添加一个内部位图，用于表示列值是否为 NULL。

    最后，表还可以通过分区表达式进行分区（范围分区、哈希分区或轮询分区）。为了实现分区裁剪，ClickHouse 还会为每个分区存储分区表达式的最小值和最大值，用于查询时跳过不需要的分区。

    1.1.2 数据裁剪#

    就算是列式存储，可以减少 IO，但是面对单词查询就要扫描 PB 级别的数据，速度和成本也难以接受，因此往往数据库还需要进一步减少扫描的数据量，这就是数据裁剪的目的。

    在 ClickHouse 中数据裁剪通过以下方式实现：

    • 主键索引（稀疏索引） 通过为表定义主键索引，来决定每个 Partition 中行的排序顺序，该索引是局部聚簇。另外分区内还会存储每个 Granule 中的首个主键列值到 Granule ID 的映射。这样查询时，如果是根据主键列查询，那么可以二分查找主键索引来快速定位。
    • 表投影 即包含相同行，但是按照不同的主键排序的表。但是很显然，这样会增加 insert、merge 和存储开销。
    • 跳过索引 这是一种比投影更轻量的方案。它旨在索引数据块（多个连续 Granule 的层级）存储少量元数据，从而避免扫描无关行。ClickHouse 中支持的跳过索引有：
      • MinMax：为每个索引数据块存储索引表达式的最小值和最大值。
      • Set：存储每个索引数据块中可配置数量的唯一值。
      • BloomFilter：针对行、token 或者 n-gram 值进行构建。

    1.1.3 合并时数据转换#

    BI 和 Observability 场景要求数据库既能支撑高数据摄取速率，又能通过聚合或数据老化等技术持续减少历史数据量。ClickHouse 提供了不同的合并策略对于已有数据进行持续的增量转换：

    • Replacing Merges：通常用作一种合并时更新机制（通常用于更新频繁的场景），或者作为插入时数据去重机制的替代方案。
    • Aggregating Merges：将具有相同 primary key 列值的多行折叠为一行聚合结果，非 primary key 列必须是汇总值。
    • Time-To-Live (TTL) Merges：为历史数据提供老化机制，可以用于将冷数据移动到成本更低的存储介质中。

    1.1.4 更新和删除#

    MergeTree* 表引擎的设计更适合仅追加（append-only）的工作负载，但某些场景下仍需要偶尔修改已有数据。

    更新时，ClickHouse 会就地重写表中的所有 parts。为了避免删除整表或者更新列时，表或者列的大小在短时间内翻倍，这个操作不提供原子性（如果要提供原子性，需要一次性生成新副本，修改完成后再替换旧的 part）。不提供原子性，意味着并行执行的 SELECT 可能同时读取到已修改和未修改的 parts。这样也可以减少修改带来的开销。

    删除时，则依然需要较大的开销，因为如果要删除特定行，则需要对表的所有列进行删除。作为替代方案，ClickHouse 提供了轻量级删除（Lightweight Delete），即维护一个位图列，用于标记某一行是否被删除。在 SELECT 查询时，针对该位图列进行过滤。

    轻量级删除可以降低删除带来的开销，但是会降低查询性能。

    1.1.5 幂等插入#

    在实际的业务场景中，客户端发送数据到服务器时，可能因为各种原因导致无法获取到执行结果（如超时、网络异常等）。这种情况下，客户端无法判断数据是否已经插入到数据库中，在传统的数据库中，往往通过唯一索引来避免重复插入。但是，对于列式数据库来说，维护这样的一个唯一索引带来的开销往往是难以接受的。

    因此，ClickHouse 提供了一种更轻量级的替代方案：服务器会维护最近 N 个已经插入 parts 的 hash 值，并忽略那些 hash 值已经存在的 parts 的重新插入。为了对去重提供更多控制，还允许客户端提供插入数据的 insert token。

    1.1.6 数据复制#

    ClickHouse 基于 Raft 协议提供分布式且具备容错的协调层。当各节点执行本地的 insert、merge、mutation 和 DDL 语句时，复制日志会在所有其他节点上异步回放。这里不关注列式数据库的分布式特性，因此也不展开。

    1.1.7 ACID 特性#

    为最大化并发读写操作的性能，ClickHouse 尽可能避免使用锁。查询在开始时会基于所有相关表中全部 parts 的一个快照来执行。这样可以保证在执行期间，由并行 INSERT 或合并操作新插入的 parts 不会参与本次查询执行。为防止 parts 被同时修改或删除，在查询执行期间，会增加这些被处理 parts 的 引用计数。

    因此，ClickHouse 从严格意义上来说并不满足 ACID，只有 SELECT 和 INSERT 并行时，能做到互不影响（INSERT 会新建一个 part，SELECT 会对 parts 产生一个快照）。

    这样的取舍在 OLAP 场景下是完全可以接受的，放弃原子性可以换来更高的读写性能。

    小结#

    ClickHouse 的存储是一个多级结构，每一级都有明确的职责：

    Table → Partition → Part → Column File (.bin) → Chunk → Granule
    • Partition（分区）：按业务维度（通常是时间）划分数据，配合 minmax_*.idx 实现分区裁剪，查询时整块跳过无关分区。
    • Part（分片）：数据写入的基本单位，不可变，后台异步合并。不可变性让并发读写无需加锁（SELECT 基于 parts 快照执行），代价是更新和删除开销大。
    • Column File（列文件）：每列独立存储，查询时只读取涉及的列；同类型数据连续排列，天然适合压缩编码。
    • Compressed Chunk（压缩块）：物理 I/O 的最小单位（~1MB），包含 Header 和多个 Granule 的压缩数据。通过 .mrk 标记文件实现 Granule 级别的随机访问而无需解压整个列文件。
    • Granule（粒度）：逻辑上的最小扫描单元（8192 行），稀疏索引和跳过索引都以 Granule 为粒度工作。粒度越小定位越精确，但索引开销越大。

    设计上的核心取舍：

    选择 获得 牺牲
    列式存储 高压缩率、列裁剪、向量化计算 点查和行读取性能差
    不可变分片（append-only） 无锁并发、简化一致性 更新/删除代价高
    稀疏索引（非 B+ 树） 索引体积小、写入无额外开销 无法精确定位单行，最小粒度是 Granule
    放弃严格 ACID 更高的读写吞吐 并发修改时无事务隔离
    Part hash 去重（非唯一索引） 写入路径轻量 只能去重整个 Part，非行级去重

    1.2 列式数据库的查询引擎#

    前面介绍过 ClickHouse 中采用了多层级存储结构，得益于这样的设计，查询时可以分三个层级对查询进行并行化，这三个层级分别是：表分片（Table Shard）、数据块（Data chunks）和数据元素（Data elements）。

    表分片，这里是指分布式架构中，将分布式表根据分片键将数据分配到不同的 Shard（位于不同的节点或者集群）上，每个 Shard 本质上都是一个独立的表。

    这一点和传统数据库的分库分表相同。

    先简单描述一下并行查询的过程：

    • 第一层级，在进行查询的时候，先根据分片键将查询请求路由到对应的 Shard 上。
    • 第二层级，在每个 Shard 上，将查询拆分为多个数据块，这些数据块可以并行处理。
    • 第三层级，在每个数据块内部，对数据元素进行并行计算。

    在支持并行计算的基础上，通常还会先进行裁剪操作，尽可能少扫描无关数据，减少查询过程中的 IO 开销。

    并行执行虽然可以提升吞吐，但是并发数不是越多越好。过多的并发会增加 CPU 上下文切换和调度开销，甚至导致资源争用，所以需要结合查询规模和硬件能力做平衡。

    从实现上看，查询通常会经历几个阶段：先根据 SQL 构建执行计划，再从执行计划中提取出涉及的表、列和过滤条件，之后由执行器把计划拆成若干子任务并发执行，最后把中间结果合并为最终结果。

    1.2.1 查询执行框架#

    在我们手搓的列式数据库中，查询执行框架可以简化为如下链路：

    • SQL Parser：负责解析 SQL，提取表名、过滤条件和投影列。
    • Query Engine：负责把查询计划拆成针对分区、分片和列的任务。
    • Storage：负责真正读取列文件、标记文件和索引文件。
    • Residual Filter：在最小扫描范围内做最终逐行判断。

    1.2.2 查询裁剪#

    查询裁剪的目标是尽可能把不相关的数据在更早的阶段排除掉。一个典型的裁剪顺序可以概括为：

    分区裁剪 → 分片裁剪 → Granule 裁剪 → 行级过滤
    • 分区裁剪：通过 minmax_*.idx 判断一个分区是否可能命中条件。
    • 分片裁剪：利用 primary.idx 或分片级元数据快速缩小范围。
    • Granule 裁剪：通过 .mrk 文件定位到需要读取的压缩块。
    • 行级过滤：在已经解压出来的少量行上执行最终条件判断。

    1.2.3 并行执行#

    在每个裁剪层级之后，都可以继续拆分任务并行执行。比如分区之间可以并行扫描，不同列也可以并发读取，再结合向量化处理，就能让查询吞吐更高。

    1.2.4 小结#

    列式数据库的查询引擎核心不是“把所有数据尽快扫一遍”，而是“尽可能少读、少解压、少比较”。

    2. 手搓一个列式数据库#

    我们基于 Go 实现一个非常简化的列式数据库，目标不是复刻 ClickHouse，而是把几个核心概念跑通：

    • Table 负责协调写入和查询
    • Partition 负责按分区聚合数据
    • Part 负责承载不可变数据文件
    • primary.idx 负责稀疏主键索引
    • .bin / .mark 负责列数据和定位信息

    2.1 存储引擎#

    存储引擎最重要的职责是把内存里的行数据变成磁盘上的不可变结构。写入路径通常会经历：

    2.1.1 写入入口#

    func (t *Table) Flush() error {
	t.mu.Lock()
	defer t.mu.Unlock()

	if err := t.flushAllLocked(); err != nil {
		return err
	}

	for _, p := range t.partitions {
		if err := p.flush(); err != nil {
			return err
		}
	}

	return t.saveTableSchema()
}

    Flush 的作用是把“内存中的临时状态”转换成“磁盘上的稳定状态”。

    2.1.2 Granule 和 Mark 文件#

    type markEntry struct {
	granuleID     uint32
	blockOffset   uint64
	rowOffset     uint32
	rows          uint32
	compressedSize uint32
}

    markEntry 记录了一个 granule 在压缩文件中的位置信息，以及解压后在块内的偏移。这样查询时就不必从头扫整列，而是可以直接跳到目标 granule 对应的压缩块。

    func encodeMarkEntry(entry markEntry) []byte {
	buf := make([]byte, 24)
	binary.LittleEndian.PutUint32(buf[0:4], entry.granuleID)
	binary.LittleEndian.PutUint64(buf[4:12], entry.blockOffset)
	binary.LittleEndian.PutUint32(buf[12:16], entry.rowOffset)
	binary.LittleEndian.PutUint32(buf[16:20], entry.rows)
	binary.LittleEndian.PutUint32(buf[20:24], entry.compressedSize)
	return buf
}

    这里的 .mark 文件，本质上就是 granule 到压缩块的映射表。

    2.1.3 分片结构#

    每个 Part 落盘后会生成一组列文件和元数据文件。对于列式存储来说，最重要的不是“文件很多”，而是“文件之间的职责清晰”。

    2.1.4 小结#

    写入路径的核心是：先写内存，再落不可变 Part；先组织列数据，再补齐索引和标记文件。这样做的好处是写入简单、并发容易处理，但代价是后续合并和删除会更重。

    2.2 查询引擎#

    查询引擎关注的重点是：尽量少读数据，尽量早裁剪掉无关内容。一个查询从 SQL 到结果，大致会经历解析、计划生成、裁剪、读取和过滤几个阶段。

    2.2.1 查询流程#

    查询可以简化为：

    • Parser 解析 SQL，拿到表名、where 条件和投影列
    • Planner 或 Engine 决定需要访问哪些 Partition、Part、Column
    • Storage 按需读取 .bin 和 .mark 文件
    • Filter 做最终逐行过滤

    2.2.2 查询裁剪链路#

    裁剪链路可以概括成：

    Partition 裁剪 → Part 裁剪 → Granule 裁剪 → Residual Filter

    对应到实现里，分区元数据、主键索引和标记文件分别承担不同粒度的缩小扫描范围职责。

    2.2.3 小结#

    查询引擎不是把所有东西都加载进来再过滤，而是尽可能把“必须处理的数据量”压到最小。

    2.3 小结#

    这篇文章从列式数据库的整体结构讲到一个简化实现，核心就是两个词：写入时不可变，查询时可裁剪。

    • 写入时：把内存状态整理成 Part，生成列文件、索引文件和标记文件
    • 查询时：先裁剪分区，再裁剪 Part 和 granule，最后才做逐行过滤

    这也是列式数据库在 OLAP 场景里性能更好的根本原因。
